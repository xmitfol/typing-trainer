# ADR-007 · OAuth-вход (Yandex + VK) — server-side authorization code flow

- **Status**: ✅ Accepted
- **Date**: 2026-06-24
- **Owner**: Клод (Architect), prep для Sprint 2 (S2.1–S2.4)
- **Related**: [03_IMPL_PLAN.md §4 Sprint 2](../03_IMPL_PLAN.md), [02_TSD.md §3.2](../02_TSD.md), модель `OAuthAccount` (backend/app/models/user.py), [ADR-006](ADR-006.md) (auth-cookies), [risks.md R-005](../risks.md)

## Context

Sprint 2 добавляет вход через **Yandex** и **VK** (S2.1–S2.4). Модель `oauth_accounts` уже в миграции (S1.1): `(provider, external_id)` уникальны, `raw_payload` JSONB, FK на `users` с cascade. Нужно зафиксировать flow, безопасность и резолвинг аккаунтов до имплементации, чтобы Борис не переделывал.

Ограничение из текущей схемы: `users.audience` и `users.character` — NOT NULL с CHECK-констрейнтами. Провайдер их не отдаёт → нужно решение, как создавать OAuth-юзера.

## Decision

**Server-side Authorization Code flow** (не implicit, не client-side). Два endpoint'а на провайдера:

```
GET /api/v1/auth/oauth/{provider}/start
    → 302 на authorize-URL провайдера (state + redirect_uri, для VK ID — PKCE)
GET /api/v1/auth/oauth/{provider}/callback?code=&state=
    → обмен code→token, fetch userinfo, резолв аккаунта, наши httpOnly cookies,
      302 на фронт (dashboard или onboarding для новых)
```

`{provider}` ∈ `yandex | vk`. Generic-слой `app/core/oauth.py` — общий интерфейс + per-provider стратегии (S2.2).

### 1. Provider strategy (generic helper)

Интерфейс стратегии: `authorize_url(state, code_challenge?)`, `exchange_code(code, code_verifier?) -> token`, `fetch_userinfo(token) -> {external_id, email, name, locale}`.

| | Yandex | VK |
|---|---|---|
| Authorize | `oauth.yandex.ru/authorize` | **VK ID** `id.vk.com/authorize` (OAuth 2.1) |
| Token | `oauth.yandex.ru/token` | `id.vk.com/oauth2/auth` |
| Userinfo | `login.yandex.ru/info` (scope `login:email login:info`) | VK ID `user_info` |
| **PKCE** | не требуется (используем standard) | **обязателен** (VK ID 2.1) |
| email | отдаёт при scope `login:email` | отдаёт при согласии (VK ID `email` scope) |

### 2. CSRF + PKCE — state в Redis

- На `/start`: генерим `state` (`secrets.token_urlsafe`), для VK — ещё `code_verifier` (PKCE). Кладём в Redis `oauth:state:{state}` → JSON `{provider, code_verifier?}`, TTL 10 мин.
- На `/callback`: `GETDEL oauth:state:{state}`. Нет/истёк → 400. Это защищает от CSRF и подделки callback. (Тот же одноразовый-через-Redis паттерн, что у email-токенов в [ADR-006]/email_tokens.)

### 3. Резолвинг аккаунта (порядок)

1. **По `(provider, external_id)`** — есть `OAuthAccount` → это вход существующего юзера. Логиним.
2. **Иначе по email** (provider email верифицирован) — есть активный `users.email == email` → **линкуем**: создаём `OAuthAccount`, ставим `email_verified=true`. (Провайдер подтвердил владение email.)
3. **Иначе** — новый юзер: создаём `User` + `OAuthAccount` + `user_settings`.

Если провайдер **не отдал email** → 302 на фронт с ошибкой `OAUTH_NO_EMAIL` (email — NOT NULL unique, без него аккаунт не создать). Просим переавторизоваться с доступом к email.

### 4. Новый OAuth-юзер и обязательные audience/character

Провайдер не даёт `audience`/`character`. Решение: создаём юзера с **провизорными дефолтами** (`audience='adult'`, `character='maxim'`, `language` из provider locale → `ru`/`en`, `email_verified=true`, `password_hash=NULL`). Callback для **нового** юзера редиректит на `onboarding.html?oauth=1` — фронт собирает audience/character и сохраняет через `PATCH /me` (Sprint 3). Существующий юзер → сразу `dashboard.html`.

Признак «новый» передаём через query редиректа (`?new=1`), без изменения схемы. До Sprint 3 (PATCH /me) OAuth-юзер живёт с дефолтами — приемлемо для MVP (он залогинен, профиль донастроит позже).

### 5. Сессия и токены провайдера

- После резолва — выдаём **наши** httpOnly cookies (access+refresh), как в `signin` (переиспользуем `_set_auth_cookies`).
- **Токен провайдера не храним** — используем один раз для `fetch_userinfo`. В `oauth_accounts.raw_payload` кладём снапшот userinfo (для аналитики/дебага), не access-token.

### 6. Config (уже есть в config.py, Sprint 2 заполнить)

```
YANDEX_OAUTH_CLIENT_ID / _SECRET
VK_OAUTH_CLIENT_ID / _SECRET
```
+ redirect_uri = `{FRONTEND_BASE_URL}/api/v1/auth/oauth/{provider}/callback` (same-origin за прокси, ADR-006/B1-003). Регистрируется в кабинете провайдера (S2.1 — PO action).

## Consequences

### Лучше
- Единый generic-слой → VK добавляется как вторая стратегия за ~0.5d (S2.4).
- Нет хранения чужих токенов → меньше surface (152-ФЗ, утечки).
- Переиспользуем cookies/Redis-one-time паттерны Sprint 1 — консистентно.

### Цена / риски
- **Account-takeover edge** (linking по email): если кто-то завёл локальный аккаунт на чужой email и не подтвердил, а владелец входит через OAuth — линк отдаст OAuth-вход к уже существующей записи. Митигация: линкуем только к активному юзеру, и provider-email верифицирован; при линке выставляем `email_verified=true`. **Сергей (Security) ревьюит перед prod** — возможно ужесточить (требовать, чтобы локальный аккаунт был email_verified, иначе создавать отдельный). Помечено остаточным риском.
- **VK ID миграция**: VK перешёл на VK ID (OAuth 2.1 + PKCE). Старый `oauth.vk.com` deprecated — стратегия VK сразу на VK ID.
- Провизорные дефолты профиля у OAuth-юзеров до Sprint 3 (PATCH /me) — косметика, не блокер.

### Триггеры пересмотра
- Sergei security-review линкинга по email до prod.
- Если провайдеры меняют userinfo-контракт — правка только в стратегии.

## Alternatives Considered
- **Client-side / implicit flow** — отклонено: токен в браузере, хуже безопасность, провайдеры депрекейтят implicit.
- **Сторонний OAuth-агрегатор (Auth0/etc)** — отклонено: лишний vendor + 152-ФЗ surface; провайдеров всего 2, generic-слой дешевле.
- **Хранить provider access/refresh токены** — отклонено: не нужны (одноразовый userinfo), лишний секрет-материал.
- **Отдельный аккаунт на каждый provider-email вместо линка** — отклонено: дублирует юзеров, плохой UX («почему мой прогресс пропал»). Линкуем по верифицированному email.
