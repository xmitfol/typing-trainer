# ADR-004 · Single-vendor infra: Yandex Cloud as code+build+host

- **Status**: ✅ Accepted
- **Date**: 2026-06-06
- **Owner**: PO (Иван), recommended by Ника (PM) + Клод (Architect)
- **Related**: [PRD §9 Risks](../01_PRD.md), [TSD §6 Deployment](../02_TSD.md), [risks.md R-001/R-011](../risks.md), [03_IMPL_PLAN.md §2](../03_IMPL_PLAN.md)

## Context

GitHub-аккаунт `xmitfol` заблокирован с ~2026-06-03 (R-001 HIGH/🔴). Локально на ветке `integration/new-shell` накопилось 37 коммитов поверх master, sprint 0 backend SDD стартовать не может (требуется remote + CI).

PO ждал ответа GitHub support 3 дня — нет. По risk-протоколу триггер эскалации на запасной remote = 7 дней. PO принял решение **не ждать**, мигрировать немедленно.

Возникает выбор vendor'а для git-репо + CI/CD:

| Опция | Verdict |
|---|---|
| Новый GitHub аккаунт (primary) | Риск повторного бана по email/IP/payment |
| GitLab.com | Solid mainstream, но удваивает vendor surface |
| Bitbucket | Слабее GitLab, отклонено |
| Codeberg (Forgejo) | CI в бете, не подходит для backend |
| **Yandex Cloud Code Repository** | Single-vendor с остальной инфрой |
| Self-hosted Gitea | Maintenance overhead, отклонено |

По [TSD §6 Deployment](../02_TSD.md) production-стек уже завязан на Yandex Cloud:
- Compute (VM 2 vCPU / 4GB)
- Managed PostgreSQL 16
- Managed Redis 7
- Object Storage (Yandex S3-compat для файлов — закрытие T4)
- Cloud Logging (закрытие T3)
- Container Registry (для Docker образов)

Email — Yandex 360 SMTP (T1). Платежи — YooKassa (тоже RU-юрисдикция, отдельная компания, но фактический ecosystem).

Single-vendor консолидация даёт: один биллинг, единый IAM, common audit log, упрощённый compliance под 152-ФЗ.

## Decision

**Primary Git remote = Yandex Cloud Code Repository.** CI/CD = Cloud Build pipeline (нативная интеграция).

**Secondary mirror = новый GitHub аккаунт** (push-only mirror через post-receive hook или scheduled `git push --mirror`). Назначение: escape hatch на случай R-011 (Yandex Cloud API breaking change) + потенциальная discoverability для open beta.

Конкретный flow CI:
```
git push → YC Code Repo (primary)
        ↓
   YC Cloud Build pipeline (lint + test + Docker build)
        ↓
   YC Container Registry (versioned images)
        ↓
   Cloud Functions / Compute Instance (prod deploy)
        ↓
   post-deploy: git push --mirror на GitHub mirror
```

## Consequences

### Что становится лучше
- **Single bill, single IAM, single audit** — упрощение compliance под 152-ФЗ
- **Нулевая миграция CI** — мы ещё не написали GH Actions yaml, пишем сразу под Cloud Build
- **Latency RU-RU** — git push быстрее с RU-сетей чем на GitHub.com
- **Mirror как escape** — снижает R-011 severity с HIGH до MED
- **Sprint 0 unblock'ается** — задача S0.5 «Git workflow» переписывается под YC, остаётся в estimate'е дня

### Цена
- **Vendor lock-in** — формализуется как принимаемый. Mitigation = mirror на GitHub + Terraform для деплоя infra (TF код переносим)
- **Меньше community-friendly** — для v1.0 не критично (RU-only маркетинг, R-009 принят)
- **YC Code Repo менее знаком команде** чем GitHub — кривая обучения ~1-2 дня для Бориса
- **Pay-as-you-go стоимость** — оценка < 300₽/мес для нашего размера; включается в общий бюджет 3000₽/мес из PRD §6.3

### Будущие шаги / триггеры пересмотра
- Если у Yandex Cloud случится API breaking change (R-011) — push-mirror всегда содержит полную историю
- Если запускаем международный маркетинг (v1.1+) — рассмотреть GitHub primary с YC mirror (обратная схема)
- Если в команду приходит OSS-контрибьютор — миграция через mirror займёт 1 день

## Alternatives Considered

### А. Ждать GitHub support ещё 7 дней
Отклонено: каждый день простоя = +1 день к дедлайну (PERT P80 = 17 октября). Парт PO принял риск повторного бана GH как меньший, чем риск потери времени.

### Б. GitLab.com primary + GitHub mirror
Отклонено: удваивает vendor surface без выгоды. У нас уже Yandex (Cloud + 360) + YooKassa, добавлять третий — без основания.

### В. Новый GitHub primary
Отклонено: повторный бан по email/payment/IP — высокий риск, если первый бан был автоматический. Mitigation в виде «купим новый IP» не масштабируется.

### Г. Self-hosted Gitea на Yandex VM
Отклонено: добавляет maintenance overhead (backup, security patches, downtime). Для solo backend team (Борис) — bus-factor усиление.
