# ADR-008 · Provider-agnostic billing abstraction

- **Status**: ✅ Accepted
- **Date**: 2026-07-01
- **Owner**: Клод (Architect), запрос PO (Иван)
- **Related**: [ADR-005](ADR-005.md) (YooKassa Hybrid renewal), `app/models/subscription.py`, [ROADMAP.md](../../../ROADMAP.md) P1, `assets/js/pricing.js`

## Context

ADR-005 зафиксировал YooKassa Hybrid recurring как механизм продления. Но:
1. **Нельзя тестировать без реального YK-shop** — верификация бизнеса 1-3 рабочих дня; на момент разработки shop не подтверждён (R-002).
2. **Хотим менять/добавлять провайдеров** (CloudPayments, Stripe для intl, СБП) без переписывания бизнес-логики.
3. **Нужен дев/тест-поток без единого провайдера** — чтобы весь путь checkout → подписка → paywall работал в CI/локально.

PO прямо сформулировал: *«универсальный биллинг, в котором потом подставим данные провайдера, и всё заработает».*

Модель `Subscription` уже есть (FSM 6 состояний), но поле заточено под `yookassa_payment_id`.

## Decision

Ввести **провайдер-агностичную абстракцию**: бизнес-логика зависит от интерфейса `PaymentProvider`, а не от конкретного провайдера. Конкретные провайдеры — плагины, выбираются конфигом.

### 1. Интерфейс `PaymentProvider` (core/billing.py)
Абстрактный протокол. Все методы работают в терминах НАШИХ доменных типов, не сырых ответов провайдера.

```
class PaymentProvider(Protocol):
    name: str
    def create_checkout(self, *, user_id, plan, period, amount_kopecks,
                        currency, idempotency_key, return_url) -> CheckoutResult: ...
    def parse_webhook(self, *, headers, raw_body) -> WebhookEvent | None: ...   # verify sig + normalize
    def charge_recurring(self, *, payment_method_id, amount_kopecks, currency,
                         idempotency_key) -> ChargeResult: ...                  # auto-renew
    def cancel(self, *, provider_payment_id) -> None: ...                       # если нужно провайдер-side
```

Доменные типы (dataclasses):
- `CheckoutResult(provider_payment_id, confirmation_url, status)` — куда редиректить юзера.
- `WebhookEvent(kind, provider_payment_id, status, amount_kopecks, payment_method_id, raw)` — нормализованное событие (`kind`: `payment.succeeded` / `payment.failed` / `payment.canceled` / `refund`).
- `ChargeResult(provider_payment_id, status, error_code, error_message)`.

### 2. Провайдеры
- **`StubProvider`** (дев/тест/CI): `create_checkout` возвращает `confirmation_url` на локальную «эмуляцию оплаты» + мгновенный success через webhook-симуляцию; `parse_webhook` доверяет подписи `stub`; `charge_recurring` всегда success. Позволяет прогнать весь путь без денег.
- **`YooKassaProvider`** (prod, по ADR-005): реальные вызовы YK API + проверка webhook-подписи. Подставляем `yookassa_shop_id`/`yookassa_secret_key` — заработает.
- **Фабрика** `get_payment_provider(settings) -> PaymentProvider` по `settings.billing_provider`.

### 3. Модель — generalize
Добавить в `subscriptions`: `provider: str` (`stub`/`yookassa`/…), обобщить платёжный id. Минимальная миграция: колонка `provider` (default по текущему env). `yookassa_payment_id` оставляем (обратная совместимость), новые провайдеры пишут в неё же как «provider_payment_id» (семантика обобщается) — либо в v1.1 переименовать. Charge-audit (`SubscriptionCharge`) без изменений.

### 4. Слои
- `core/billing.py` — интерфейс + доменные типы + StubProvider + фабрика (+ каркас YooKassaProvider, тело — TODO под реальный shop).
- `services/billing_service.py` — оркестрация: `create_checkout` (создаёт `Subscription(status=pending)` + provider checkout), `apply_webhook(event)` (FSM: pending→active / failed; пишет `SubscriptionCharge`; идемпотентно по `idempotency_key`/`provider_payment_id`), `cancel_subscription` (status=cancelled, до end-of-period), `has_active_subscription(user_id)` (гейт).
- `api/v1/billing.py`:
  - `GET  /billing/plans` — каталог тарифов (публично).
  - `POST /billing/checkout` — auth; создаёт checkout, отдаёт `confirmation_url`.
  - `POST /billing/webhook/{provider}` — публичный callback провайдера → `apply_webhook`.
  - `GET  /billing/subscription` — auth; текущий статус/срок.
  - `POST /billing/subscription/cancel` — auth.
- `schemas/billing.py` — pydantic-контракты.
- `config.py` — `billing_provider: Literal["stub","yookassa"]="stub"`, `billing_currency="RUB"`, `yookassa_shop_id: str|None`, `yookassa_secret_key: SecretStr|None`, каталог планов (или в коде).

### 5. Paywall (feature-gating)
Зависимость FastAPI `require_active_subscription` (или мягкий `subscription_status`), опирается на `has_active_subscription`. Что гейтить (весь курс / после урока N / отдельные тиры) — решение PO/Полины, эндпоинты уже готовы отдавать статус.

## Consequences
- ✅ Весь платёжный путь тестируется через StubProvider (CI/локально/демо) без реального провайдера.
- ✅ Подключение YK/CloudPayments/Stripe = реализовать один класс + подставить креды, бизнес-логика не меняется.
- ✅ ADR-005 (Hybrid renewal) сохраняется: recurring/reminder — поверх абстракции (`charge_recurring` + FSM grace).
- ⚠️ Тонкость provider-webhook-подписей: у каждого своя проверка — инкапсулируется в `parse_webhook`.
- ⚠️ Реальный YooKassaProvider требует VM/YK-shop для интеграционного теста (как auth в Sprint 1).

## Rollout
1. (этот шаг) core-абстракция + StubProvider + сервис + эндпоинты + миграция + schemas → путь работает на Stub.
2. Фронт: `pricing.js` → `POST /billing/checkout` → redirect; экран статуса подписки.
3. YooKassaProvider (реальные вызовы) — когда PO подтвердит shop; интеграционный тест на VM.
4. Recurring cron (ARQ) + email-reminder (ADR-005) — отдельный шаг.
