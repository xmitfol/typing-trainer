# ADR-001 · Anonymous guest lifecycle = 3 дня

- **Status**: ✅ Accepted
- **Date**: 2026-06-06
- **Owner**: PO (Иван)
- **Related**: [PRD §8 Q4](../01_PRD.md), [TSD §4.2](../02_TSD.md)

## Context

В типичном SaaS-онбординге есть две стратегии работы с гостем:

1. **Force signup** — без регистрации ничего не работает. Высокий барьер, ниже conversion.
2. **Anonymous progressive** — гость может пройти N шагов (онбординг + 1-2 урока) в localStorage, потом promt'ится на signup. Низкий барьер, риск потери данных при долгой задержке.

Фронт сейчас (v1.0) поддерживает anonymous flow: онбординг → dashboard → урок → task — всё работает в localStorage без аккаунта. При signup'е данные мигрируют в БД через `POST /auth/migrate-guest`.

Вопрос: **как долго хранить такие гостевые данные на бекенде, если юзер не залогинился?**

Гость не имеет аккаунта — он идентифицируется только по `guest_id` (UUID в localStorage) и не зарегистрирован в `users`. Но мы хотим:
- Принимать события аналитики (`POST /events/batch`) для measure conversion
- Хранить минимальный footprint для отчётов
- НЕ накапливать «висящие» сессии бесконечно

Варианты TTL: 1ч / 24ч / 3 дня / 7 дней / 30 дней / forever.

## Decision

**Гостевые сессии (без `user_id`) хранятся 3 дня.** Daily cron очищает `events` строки с `user_id IS NULL` и `created_at < now() - interval '3 days'`.

Конкретно:
- Анонимные events живут 72 часа
- Если гость залогинился — `session_id` сохраняется в `users.last_guest_session_id`, привязка к user_id для всей будущей аналитики
- Cron `cleanup_anonymous_sessions` запускается в 03:00 MSK ежедневно через APScheduler

## Consequences

### Что становится лучше
- БД не растёт безгранично от ботов и crawler'ов
- Funnel'ы измерим: гость → signup имеет понятное окно
- Меньше PII-риск (3 дня — недостаточно для накопления sensitive паттернов)

### Цена
- Юзер, который зашёл в субботу и вернулся в среду — потеряет аналитический trail. Окей: 3 дня покрывает выходные.
- Гости, которые залогинились через >3 дня после первой сессии, не получат attribution «откуда пришли». Окей: не критично для v1.0.

### Будущие шаги / триггеры пересмотра
- После 100 paid юзеров проверить distribution time-to-signup. Если 80% укладываются в 3 дня — TTL норм.
- При росте трафика рассмотреть partitioning `events` по месяцам (вместо ежедневного DELETE)

## Alternatives Considered

### Альтернатива А — 1 час
Отклонено: слишком короткое окно. Юзер залогинится после обеда → потерял trail.

### Альтернатива Б — 30 дней
Отклонено: 152-ФЗ риск. Долго хранить identifiable patterns анонимов = persistent footprint, нужно дополнительное согласие.

### Альтернатива В — навсегда (event sourcing)
Отклонено: storage не бесконечен, growth непредсказуем (10K ботов могут залить БД).
