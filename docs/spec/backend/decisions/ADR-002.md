# ADR-002 · Profile mutation policy

- **Status**: ✅ Accepted
- **Date**: 2026-06-06
- **Owner**: Клод-архитектор (по делегированию от PO «продумай сам»)
- **Related**: [PRD §8 Q5](../01_PRD.md), [TSD §4a](../02_TSD.md)

## Context

Юзер регистрируется в типичной для онбординга комбинации `(name, audience, character, language)`. Через месяц-полгода может захотеть:

- Сменить наставника (надоела Анна, хочу Максима)
- Сменить язык курса (русский → английский)
- «Взрослеет» и хочет switch с teen на adult tier
- Случайно выбрал не тот character и хочет переиграть с нуля

Backend должен ответить на 3 связанных вопроса:

1. **Что разрешать менять, что нет?** (security: нельзя «омолодить» аккаунт)
2. **Как считать прогресс при смене tier'а?** (потерять / сохранить per-tier)
3. **Какой UX-контракт с фронтом?** (нужен ли confirm-modal, чему фронт верит)

Существующая фронт-логика: `pickTier(profile) = f(language, character)` — производное поле. Прогресс сейчас в localStorage keyed by lesson_num внутри одного tier'а (не учитывает switching).

## Decision

### Что разрешено менять

| Поле | Свободно | Триггер ограничений |
|---|---|---|
| `name`, `gender` | ✅ | — |
| `character` | ✅ | Может сменить tier — UI показывает warning «Тренируешься в другом курсе» |
| `language` | ✅ | Меняет tier (ru↔en). UI делает full re-render |
| `audience: kid → teen` | ✅ | После confirm-modal |
| `audience: teen → adult` | ✅ | После confirm-modal |
| `audience: adult → teen/kid` | ❌ | 422 `AGE_DOWNGRADE_FORBIDDEN` |
| `audience: teen → kid` | ❌ | 422 |
| `email` | ⚠️ | Двух-шаг: confirm на старый email, верификация нового |
| `password` | ⚠️ | Требует `current_password` |
| Sub-account `audience` | Parent-only | Меняет родитель + только если у него family-подписка |

### Прогресс при смене tier'а

**Сохраняется per-tier.** `progress` keyed by `(user_id, tier, lesson_num)`. Если юзер был на tier1 lesson 30, поменял character на knopych (→ ru_teen) — старый прогресс tier1 остаётся в БД. Вернётся к anna — увидит свои 30 уроков с звёздами.

**В UI** на Profile/Statistics показываем все tier'ы где есть прогресс. На Dashboard — только текущий tier (но «Архив моих курсов» доступен).

### UX-контракт

- `PATCH /me` возвращает `tier_changed: bool` + `old_tier`/`new_tier` в response
- Фронт по флагу `tier_changed=true` re-fetch'ит `/me/progress` и обновляет dashboard
- Для опасных мутаций (`audience: kid → adult`) frontend показывает confirm-modal перед запросом
- Для `email`/`password` — отдельные endpoints, не через общий `PATCH /me`

## Consequences

### Что становится лучше
- Юзер не «застревает» в одном tier — может попробовать другой контент без потери
- Прогресс защищён: эксперимент с character не убивает звёзды старого курса
- Security: нельзя обойти family-плана через смену audience у sub-account'а
- Email change protected от account takeover

### Цена
- Storage растёт: юзеры с 3 tier'ами имеют 3x progress-rows. Но это <1KB на tier, не критично
- Сложнее статистика — нужно правильно агрегировать «activity in tier X vs tier Y»
- UI complexity — Profile/Statistics показывает несколько курсов, а не один (но это полезно)

### Будущие шаги / триггеры пересмотра
- Если 50%+ юзеров никогда не меняют tier — упростить UI и убрать «архив»
- Если юзеры жалуются «потерял прогресс при смене» — возможно сделать опцию «слить два tier'а» (merge tool)

## Alternatives Considered

### Альтернатива А — менять нельзя ничего после регистрации
Отклонено: токсичный UX, юзеры будут делать новые аккаунты вместо смены наставника.

### Альтернатива Б — менять можно всё, но прогресс **сбрасывается** при смене tier'а
Отклонено: deal-breaker для retention. Юзер прошёл 30 уроков, попробовал EN, не понравилось, вернулся к RU — потерял 30 уроков. Никто не возвращается.

### Альтернатива В — `audience` нельзя менять вообще
Отклонено: дети взрослеют. Подросток в 14 лет хочет полную версию в 18. Блокировать — терять долгосрочных клиентов.
