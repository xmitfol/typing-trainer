# ADR-005 · YooKassa recurring fallback strategy (Hybrid)

- **Status**: ✅ Accepted

- **Owner**: Клод (Architect), recommended by Ника (PM)
- **Related**: [risks.md R-002](../risks.md), [03_IMPL_PLAN.md Sprint 6-7](../03_IMPL_PLAN.md), [01_PRD.md §6.3 Pricing](../01_PRD.md), [02_TSD.md §5 Payments](../02_TSD.md)
- **Supersedes**: [ADR-005-DRAFT.md](ADR-005-DRAFT.md) (Ника, prep)

## Context

R-002 в risk register: «YooKassa может отклонить recurring-схему в РФ». Условия неопределённости:
- YK поддерживает «автоплатежи» через `save_payment_method=true`, но
- Не все банки-эмитенты поддерживают сохранение карты (особенно МИР некоторых банков)
- Регуляторика (152-ФЗ + ЦБ) может изменить требования к согласию на auto-renewal в любой момент
- На момент SDD у нас **нет** подтверждения, что наш конкретный YK shop сможет включить recurring (нужна верификация бизнеса + подача заявки → 1-3 рабочих дня ожидания)

**Bus-impact** (от Полины):
- Без recurring: LTV падает на 40-60% (юзер обязан помнить + click)
- С recurring, который потом не получится включить: 3-5 дней delay в Sprint 7 на срочную переделку

PRD §6.3 предполагает месячную/годовую подписку с скидками до -35%. ADR-005 определяет **как реализуем продление**.

## Decision

**Hybrid (Вариант В из draft'а)**: пробуем recurring, при отказе YK или per-user failure — автоматический fallback на email-reminder с manual-click flow. Оба механизма всегда live, переключение по флагу `subscription.is_auto_renew`.

### Ключевые правила

1. **Default at checkout**: сохраняем `payment_method_id`, ставим `is_auto_renew=true`
2. **Daily renewal cron** (ARQ worker) проверяет подписки с `next_billing_date <= today`:
   - Если `is_auto_renew=true` И `payment_method_id` есть → попытка charge через YK
   - При ошибке → автоматический switch на `is_auto_renew=false` + email-reminder flow
3. **Email-reminder flow** работает как fallback и как self-service (можно вручную выключить auto-renew)
4. **Grace period 3 дня**: если recurring fail на day-of — даём 3 дня попробовать снова + email reminders. Content lock на day 4.
5. **Cancel by user**: действует в конце текущего period'а, не immediate.

## Answers to open questions (Клод, после draft'а Ники)

### 1. APScheduler vs ARQ для renewal cron

**ARQ** (не APScheduler). Renewal — это тяжёлая (N HTTP calls к YK + DB writes + email) + критичная (любой fail = upset customer) задача. ARQ даёт встроенный retry + exponential backoff + idempotency через unique `job_id`. APScheduler in-process остаётся для лёгких задач (cleanup анонимов 3д из ADR-001, daily reports).

Очередь — отдельная (`queue=renewals`), чтобы не конкурировать с email-задачами и не блокировать друг друга.

### 2. Retry policy при YK transient errors

3 попытки с exponential backoff:
- Attempt 1 → immediate
- Attempt 2 → +5 секунд (если 5xx / network timeout)
- Attempt 3 → +25 секунд

После 3-х fail'ов:
- Email юзеру «не получилось списать, проверьте карту»
- `is_auto_renew → false`
- `last_charge_error` сохраняется в БД для analytics

**Никаких human-in-the-loop** на каждую ошибку — слишком expensive. Алерт оператору только если **>5% renewal'ов за день** fail'нулись (panic level — может быть проблема на стороне YK).

### 3. Email-cron vs trigger-based

**Cron daily-scan** (не event-driven). Простота > flexibility:
- Sprint 8 (analytics) уже использует daily-cron pattern
- Изменение сроков (7 дней → 5 дней) = изменение константы, нет cascade'а
- Event-driven (`schedule_email_at(t)`) добавляет state (поссылочные jobs) — сложнее testing

Cron работает в 09:00 MSK ежедневно:
- Подписки с `expires_at <= now + interval '7 days'` → шлём первое напоминание
- Подписки с `expires_at <= now + interval '1 day'` → шлём финальное напоминание
- Идемпотентность через `last_reminder_sent_at` колонку

### 4. Idempotency key для recurring charge

Формат: `recurring-{subscription_id}-{billing_period_iso}` (например `recurring-7d2e0bfb-2026-07`).

YK хранит idempotency key 24ч. Если retry попадает на следующий день — генерируем новый ключ `recurring-{sub_id}-{period}-r{N}` где N = retry counter. Дубль не возникает потому что:
- Перед каждым charge проверяем `subscription_charges.attempted_at` для этого period'а
- YK side: тот же idempotency key вернёт кэшированный результат в течение 24ч
- Наш side: следующий day-cron пропустит подписку с `last_successful_charge_at` в этом период'е

### 5. Schema decision

**Одна таблица `subscriptions` с FSM + отдельная `subscription_charges` для audit log.**

```sql
ALTER TABLE subscriptions ADD COLUMN:
    is_auto_renew         BOOLEAN NOT NULL DEFAULT TRUE;
    payment_method_id     VARCHAR(64);                  -- YK payment_method_id
    last_charge_attempt_at TIMESTAMPTZ;
    last_charge_error     VARCHAR(255);                  -- code или null если success
    last_reminder_sent_at TIMESTAMPTZ;
    grace_until           TIMESTAMPTZ;                   -- 3 дня после fail'а

-- FSM status расширяется:
-- 'pending' | 'active' | 'grace' | 'expired' | 'cancelled' | 'failed'

CREATE TABLE subscription_charges (
    id                  BIGSERIAL PRIMARY KEY,
    subscription_id     UUID NOT NULL REFERENCES subscriptions(id),
    yookassa_payment_id VARCHAR(64) UNIQUE,
    idempotency_key     VARCHAR(128) NOT NULL,
    attempted_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    status              VARCHAR(20) NOT NULL,            -- 'success' | 'failed' | 'pending_3ds'
    amount_kopecks      INTEGER NOT NULL,
    error_code          VARCHAR(64),
    error_message       VARCHAR(512),
    is_recurring        BOOLEAN NOT NULL DEFAULT FALSE,  -- vs manual click
    metadata            JSONB NOT NULL DEFAULT '{}'::jsonb
);
CREATE INDEX idx_sub_charges_sub ON subscription_charges(subscription_id, attempted_at DESC);
```

Audit для всех попыток — в `subscription_charges`. Текущий статус подписки — в `subscriptions`. FSM transitions логируются в `events` (Sprint 8).

### 6. Cancel flow

**В конце текущего period'а**, не immediate. Reasoning:
- UX: юзер уже оплатил → пусть дослужит до конца
- Revenue нейтрально: деньги уже получены
- Меньше «обиженных»: «я оплатил а вы выключили сразу — верните деньги»

При `POST /me/subscriptions/{id}/cancel`:
- `is_auto_renew → false`
- `cancelled_at = now()` (для analytics)
- `expires_at` остаётся прежним
- Email-чек «подписка отменена, действует до X»
- В day-of expiry: standard expiry flow

Возврат денег (refund) — отдельная фича, не v1.0.

### 7. Grace period

**3 дня** (по industry standard SaaS). Если recurring fail на `next_billing_date`:
- `status → 'grace'`
- `grace_until = next_billing_date + interval '3 days'`
- В течение grace: контент доступен, email-reminder каждый день
- После grace: `status → 'expired'`, content lock

В PRD §8a (открытые v1.1+) пометил «возможный апгрейд: grace period adjustable per pricing tier».

### 8. i18n emails

**RU+EN с самого начала**. Reasoning:
- Cost negligible — просто 2 шаблона вместо 1
- Hedge на случай если EN маркетинг откроется (R-009 принят но не вечно)
- Юзер с `profile.language='en'` ожидает английские письма (UX consistency)

Структура: `app/core/emails/templates/<event>/{ru,en}.jinja2`. Selector по `user.language`.

### 9. YK SCA / 3DS при recurring

Если YK возвращает `requires_action` (3DS challenge нужен) — recurring **не может пройти без юзера**. Действие:
- `last_charge_error = 'requires_action'`
- `is_auto_renew → false`
- Email с темой «Требуется подтверждение карты»: CTA на manual checkout
- Юзер кликает → YK redirect с SCA → если успех, можем re-enable auto_renew (доп. ADR в v1.1)

Не пытаемся automation SCA — это противоречит SCA-протоколу.

### 10. Analytics events

Добавляем в Sprint 8 (S8.4-S8.5) события:
- `payment_method_saved` / `payment_method_save_failed` — на initial checkout
- `subscription_renewed_auto` — успешный recurring charge
- `subscription_renewed_manual` — manual click через email-flow
- `subscription_renewal_failed` (с reason) — после 3 retry'ев
- `subscription_renewal_recovered` — fallback на manual после ошибки

Funnel queries в `analytics-queries.sql`:
- Conversion `renewed_auto / due_for_renewal` (recurring success rate)
- Conversion `renewed_manual / renewal_failed` (recovery rate)
- Time-to-recovery (от fail до manual click)

## Consequences

### Что становится лучше
- **Best LTV** для cohort'а с работающим recurring (90-100% retention vs ~50% pure manual)
- **Resilient**: даже если YK откажет shop-wide — код уже умеет работать в режиме reminder без переделки
- **Раннее знание** через T-001…T-003: на конец Sprint 6 знаем точно, recurring доступен или нет
- **Лёгкое A/B testing**: forced-disable recurring для cohort'а → сравнение retention
- **Regulatory hedge**: оба механизма live, переключение по флагу

### Цена
- **+1d в Sprint 7** (2.5d вместо 1d на S7.1) — учтено в impl plan update ниже
- **+0.5d в Sprint 8** на analytics split (отдельные events)
- **Сложнее testing**: 4 ветки (recurring success / recurring fail → fallback / manual click / cancel)
- **Email volume больше**: те, у кого recurring работает, всё равно получают charge-confirmation (минор UX cost)
- **Schema более сложная**: `subscriptions` + `subscription_charges` audit log

### Будущие шаги / триггеры пересмотра
- После Sprint 6 day 5 (T-003) — финальное решение «hybrid или fallback на Б»
- После week 12 beta — review decline rate. Если > 30% — пересмотр.
- При изменении 152-ФЗ — возможен revert на Variant Б (no auto-renew = no extra согласие)
- Если YK breaks recurring API — fallback автоматический (`is_auto_renew → false` для всех)

## Alternatives Considered

### А. Pure recurring (auto-renewal только)
**Отклонено**: Risk YK refusal слишком велик. При срабатывании = 3-5 дней emergency rework в Sprint 7. Hybrid даёт ту же upside + защиту от риска за +1d cost.

### Б. Single-payment + email-reminder (manual only)
**Отклонено**: LTV drop 40-60% — недопустимо для unit economics. Если recurring работает у YK — этот вариант оставляет деньги на столе.

Однако **Б остаётся fallback'ом для MVP** если T-003 сработает (recurring не одобрен к концу Sprint 6).

### Г. Defer payment integration до Sprint 8
**Отклонено**: ломает gate Sprint 6/7 в IMPL_PLAN. PO-цель: 10 paying users к open beta (week 11-12) — без payment integration не достижимо.

### Д. Сторонний биллинг (Lava / Robokassa)
**Отклонено**:
- Усложняет vendor surface (ещё один partner, IAM, billing)
- 152-ФЗ surface увеличивается (cross-vendor data flow)
- YK — стандарт для RU, есть поддержка Yandex Pay + СБП «из коробки»

### Е. Custom recurring через сохранённую карту YK API (без формального "автоплатёж" flag)
**Отклонено**: 152-ФЗ требует явного согласия на recurring. Manual «scheduled» charge без согласия = регуляторная мина. Hybrid использует formal autopay механизм YK, который compliant by design.
