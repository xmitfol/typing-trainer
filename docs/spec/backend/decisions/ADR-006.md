# ADR-006 · Self-hosted anti-bot вместо Yandex SmartCaptcha

- **Status**: ✅ Accepted
- **Date**: 2026-06-11
- **Owner**: Клод (Architect), решение PO (Иван)
- **Related**: [risks.md R-007](../risks.md), [security/argon2_and_captcha.md §2](../security/argon2_and_captcha.md), [03_IMPL_PLAN.md Sprint 1](../03_IMPL_PLAN.md), [backend/app/core/captcha.py](../../../../backend/app/core/captcha.py)
- **Supersedes**: выбор провайдера в `security/argon2_and_captcha.md §2` (Yandex SmartCaptcha → self-hosted). Argon2-часть (§1) той же спеки **не затрагивается**.

## Context

R-007 в risk register: «бот-регистрация» — массовое создание фейковых аккаунтов забивает БД, жжёт email-квоту, портит аналитику. Security spec §2 (2026-06-07) выбрала внешний провайдер **Yandex SmartCaptcha** как mitigation.

**Решение PO (2026-06-11)**: капчу делаем сами, без внешнего провайдера.

Мотивация PO:
- Не заводить ещё одну внешнюю регистрацию/зависимость ради MVP-защиты (PO-action «зарегистрировать SmartCaptcha shop» отменяется).
- Полный контроль над поведением и порогами, без обращения к стороннему API на каждый signup.
- Нулевой 152-ФЗ surface (никакой передачи IP/поведения юзера третьей стороне).

Архитектор подтверждает, что для целевой нагрузки v1.0 (rate-limit 3 signup/IP/час, не социальная сеть) self-hosted стек закрывает R-007 на приемлемом уровне. Трейд-офф (нет ML-сети, видящей ботнеты глобально) принят — см. Consequences.

## Decision

Заменяем внешний SmartCaptcha на **stateless self-hosted anti-bot из трёх слоёв**. Внешних вызовов нет — всё считается на нашем backend по HMAC-подписанному challenge.

### Слой 1 · Honeypot (бесплатно, ловит примитивных ботов)

Скрытое поле формы (`display:none`, `aria-hidden`, `autocomplete=off`, нейтральное имя вроде `nickname2`). Человек его не видит и оставляет пустым; наивный бот, парсящий все `<input>`, заполняет. Backend: **поле непустое → 403, мгновенный reject** до любой дорогой работы.

### Слой 2 · Proof-of-Work (hashcash, основной слой)

Делает массовую регистрацию вычислительно дорогой, оставаясь незаметной для одиночного человека.

**Issue challenge** — `GET /api/v1/auth/challenge`:
```json
{
  "challenge": "<base64url(payload)>",
  "signature": "<hmac-sha256(jwt_secret, payload)>",
  "difficulty": 18,
  "algorithm": "sha256-leading-zero-bits"
}
```
`payload = {"salt": <16 random bytes b64>, "iat": <unix>, "exp": <iat+600>, "difficulty": 18}`.
Stateless: подпись HMAC-SHA256 на `jwt_secret_key` (уже есть, ≥32 chars). Сервер **не хранит** выданные challenge — аутентичность проверяется подписью.

**Solve (frontend)** — найти `nonce` (счётчик), при котором `sha256(salt + ":" + nonce)` имеет ≥ `difficulty` ведущих нулевых бит. Это ~`2^difficulty` хешей. На submit форма шлёт `{challenge, signature, nonce}` + honeypot-поле.

**Verify (backend)**, дёшево и stateless:
1. HMAC `signature` валиден для `challenge` → не подделан/не подменён difficulty.
2. `now < exp` → challenge свежий (TTL 10 мин).
3. `sha256(salt + ":" + nonce)` действительно имеет ≥ `difficulty` нулевых бит → PoW решён.
4. **Replay-guard**: `salt` кладётся в Redis с TTL = остаток жизни challenge; повторный `salt` → 403. Один challenge = одна регистрация. (Redis уже в стеке для rate-limit.)

### Слой 3 · IP rate-limit (без изменений)

Уже в плане (slowapi + Redis): `POST /auth/signup` — 3/IP/час, `POST /auth/signin` — 5/IP/мин. Капча — **дополнение**, не замена.

### Где применяется (без изменений к таблице spec §2)

| Endpoint | Защита |
|---|---|
| `POST /auth/signup` | honeypot + PoW (always) |
| `POST /auth/forgot` | honeypot + PoW (always) |
| `POST /me/family/add` | honeypot + PoW (always) |
| `POST /auth/signin` | conditional: PoW требуется после 3 failed/IP/час |
| `POST /events/batch` (аноним) | conditional: при rate-limit overflow |

### Параметры (config)

```bash
CAPTCHA_POW_DIFFICULTY=18          # ведущих нулевых бит; 18≈0.1-0.5s в браузере
CAPTCHA_CHALLENGE_TTL_SECONDS=600  # время жизни challenge
CAPTCHA_HONEYPOT_FIELD=nickname2   # имя скрытого поля
# HMAC-подпись challenge использует существующий JWT_SECRET_KEY (отдельный секрет не вводим)
```

`difficulty` — рантайм-рычаг: при всплеске бот-атаки поднимаем 18→22 (×16 стоимость для бота) без передеплоя кода, только env + рестарт. Sprint 8 (behavioral analytics) может делать это адаптивно.

### Dev bypass (без изменений в семантике)

В `app_env=dev` токен `test-bypass-captcha` пропускает проверку — нужно verify-скриптам для создания test-юзеров. Сохраняется как и раньше.

## Consequences

### Что становится лучше
- **−1 внешняя регистрация** для PO, −1 runtime-зависимость, −1 точка отказа (нет «Yandex API down → signup down»).
- **Нулевой 152-ФЗ surface** по капче — поведение/IP юзера никуда не уходят.
- **Бесплатно навсегда** (нет лимита 10K/мес), полный контроль порогов.
- **Stateless** challenge (HMAC) — горизонтально масштабируется без shared state, кроме короткоживущего replay-guard в Redis (уже есть).

### Цена / риски
- **Нет глобальной ML-сети.** SmartCaptcha/reCAPTCHA видят ботнеты по всей своей сети; наш PoW видит только стоимость. Замотивированный атакующий с дешёвым CPU может платить PoW-цену. **Mitigation**: rate-limit по IP + difficulty-рычаг + слой 3 behavioral analytics в Sprint 8. Для целей v1.0 (не масс-таргет) — приемлемо.
- **PoW грузит CPU клиента** на ~0.1-0.5s. Для одной регистрации незаметно; на слабых устройствах при difficulty>20 ощутимо → держим default 18, поднимаем только под атакой.
- **Требует frontend-воркера.** PoW считаем в Web Worker, чтобы не морозить UI-поток. +небольшая работа Алексу в S1.4 (вместо вставки готового Yandex-виджета).
- **Реверсивно**: если self-hosted окажется недостаточно под реальной атакой — можно вернуть внешний провайдер новым ADR (supersede). Контракт endpoint'а (`captcha_token`/challenge) проектируем провайдер-агностично, чтобы свап был дешёвым.

### Триггеры пересмотра
- Если после launch bot-signup rate (пойманные + проскочившие) > 5%/день при difficulty=22 — рассматриваем возврат внешнего провайдера (новый ADR).
- Sprint 8: подключить captcha-метрики (challenge issue rate, PoW-solve p95, honeypot catches, replay rejects) в Grafana.

## Alternatives Considered

### А. Оставить Yandex SmartCaptcha (статус-кво спеки)
**Отклонено PO**: лишняя внешняя регистрация и зависимость ради MVP-защиты; PO хочет контроль и нулевой внешний surface.

### Б. Только honeypot + rate-limit (без PoW)
**Отклонено**: honeypot ловит лишь примитивных ботов; rate-limit по IP обходится пулом прокси/IPv6-подсетями. Без PoW массовая регистрация остаётся дешёвой. PoW добавляет стоимость, не зависящую от числа IP.

### В. Только PoW (без honeypot)
**Отклонено**: honeypot почти бесплатен и отсекает мусорный трафик до вычисления PoW — дешёвый первый фильтр. Слои дополняют друг друга.

### Г. CAPTCHA-картинки собственной генерации (распознай текст/цифры)
**Отклонено для v1.0**: ломает UX (особенно дети — наша ЦА), современный OCR/ML решает такие картинки лучше людей, нужен accessibility-fallback. PoW невидим для пользователя.

### Д. Cloudflare Turnstile / hCaptcha
**Отклонено**: те же возражения, что к внешнему провайдеру (доп. зависимость + не-RU юрисдикция / 152-ФЗ surface). Уже отклонены в spec §2.
