## 1. Цель и контекст

### 1.1. Mission

Дать команде **внутренний инструмент управления** живым SaaS: поддержка пользователей, операции с подписками и платежами, продуктовая аналитика — без прямых запросов в БД руками.

### 1.2. Зачем сейчас

Бэкенд P1 (биллинг, ADR-008) и P2 (auth, OAuth, серверный синк прогресса) готовы и прогнаны на реальном стеке. Как только пойдут живые юзеры и платежи, без админки невозможно:
- Ответить на обращение в поддержку (посмотреть аккаунт/прогресс/подписку, войти под юзером).
- Вернуть деньги / выдать или отменить подписку руками (спор, гудвилл, тех-сбой).
- Заблокировать нарушителя, сбросить пароль, подтвердить email вручную.
- Увидеть, где юзеры отваливаются, какие уроки сложные, какая конверсия в оплату и MRR.

Сейчас единственный «админ-инструмент» — `adminer` (dev-браузер БД, не для прода) + запланированные события аналитики. Модель `User` **не имеет роли**, RBAC нет.

### 1.3. Положение в roadmap

```
P1 Биллинг (ADR-008)         ✅ scaffold зелёный на стеке (gate 6/6)
P2 OAuth + серверный синк     ✅ бэкенд зелёный (OAuth 26/26, /me/* 45/45)
── Админка (этот PRD) ──────  🟡 спека → реализация по фазам
P3 Онбординг-редизайн         🟡 ждёт макета PO
P4 Адаптив / удержание        ⚪ ждёт данных
```

Админка — надстройка над готовыми доменными сервисами (`billing_service`, `auth_service`, `me_service`), а не новая подсистема.

## 2. Пользователи админки и матрица прав

Админку используют **сотрудники**, не конечные юзеры. Три роли (закреплено PO):

| Роль | Кто | Назначение |
|---|---|---|
| **analyst** | Продакт/маркетинг | Только чтение статистики и агрегатов. Не видит перс.данные точечно. |
| **support** | Поддержка | Работа с юзерами (просмотр, блок, сброс пароля, верификация, имперсонация) + отмена/выдача подписки. **Без возвратов и управления ролями.** |
| **superadmin** | Владелец/техлид | Всё, что может support + **возвраты денег** + **назначение ролей** сотрудникам. |

Роль хранится на `User` (поле `role`, дефолт — обычный юзер). Обычные юзеры (99.99%) роли не имеют и в админку не входят.

### 2.1. Матрица «роль × право»

| Право | analyst | support | superadmin |
|---|:--:|:--:|:--:|
| Смотреть обзор/статистику | ✅ | ✅ | ✅ |
| Список/поиск/карточка юзера | — | ✅ | ✅ |
| Блок/разблок (soft-delete/restore) | — | ✅ | ✅ |
| Сброс пароля (отправить письмо) | — | ✅ | ✅ |
| Ручная верификация email | — | ✅ | ✅ |
| Имперсонация («войти как юзер») | — | ✅ | ✅ |
| Отмена подписки | — | ✅ | ✅ |
| Ручная выдача/продление подписки | — | ✅ | ✅ |
| **Возврат денег (refund)** | — | — | ✅ |
| **Назначение ролей сотрудникам** | — | — | ✅ |

Чувствительные операции (возврат, смена роли, имперсонация) требуют **повторного ввода пароля** (re-auth) и пишутся в аудит-лог.

## 3. Use cases

### 3.1. Поддержка разбирает обращение
```
Обращение «не открывается урок 6» → support открывает Пользователи →
поиск по email → карточка (видит: подписки нет, урок 6 за paywall) →
имперсонация «войти как юзер» → воспроизводит → объясняет/выдаёт триал
```
**Acceptance:** support находит юзера за <30с, видит его подписку/прогресс, может войти под ним; вход зафиксирован в аудите.

### 3.2. Возврат денег
```
Юзер просит возврат → superadmin открывает Платежи → находит подписку →
видит charge-лог → «Вернуть деньги» → re-auth → refund через провайдера →
подписка/чардж помечены, аудит записан
```
**Acceptance:** возврат вызывает refund у провайдера (stub эмулирует, YooKassa — реально), создаёт charge-запись `refunded`, пишет аудит; повтор идемпотентен.

### 3.3. Аналитик смотрит воронку
```
analyst открывает Статистику → воронка signup→active→subscribed→churned
за период → видит drop-off на уроке N → отдаёт продукту
```
**Acceptance:** цифры считаются из `events`/`subscriptions`/`progress`, обновляются за период, не требуют SQL руками.

### 3.4. Superadmin выдаёт роль сотруднику
```
Новый сотрудник поддержки → superadmin открывает карточку его юзера →
«Назначить роль → support» → re-auth → аудит
```
**Acceptance:** роль меняется только superadmin'ом, с re-auth и аудитом; нельзя понизить самого себя случайно (защита).

## 4. Функциональные требования (по вкладкам)

### 4.1. Обзор (Overview)
Дашборд ключевых метрик одним взглядом (за выбранный период):
- Активные юзеры (DAU/MAU), новые регистрации.
- Активных подписок, MRR (сумма активных / период), конверсия signup→оплата.
- Мини-графики динамики (регистрации, подписки) за 30 дней.

**Acceptance:** страница грузится <2с, цифры сходятся с БД, есть переключатель периода (7/30/90 дней).

### 4.2. Пользователи (Users)
- **Список/поиск/фильтр:** поиск по email/имени/id; фильтры — audience (adult/teen/kid), email_verified, наличие активной подписки, deleted (заблокированные).
- **Карточка юзера:** профиль (email, имя, audience, character, language, verified, создан), подписки (все, со статусами), прогресс по тирам (звёзды/уроки), последние попытки, family-связи (ADR-003), OAuth-провайдеры.
- **Действия (support+):** блок (soft-delete `deleted_at`) / restore; отправить сброс пароля (письмо через `email_service`); ручная верификация email; имперсонация; открыть родителя/детей family.

**Acceptance:** поиск возвращает результат за <1с на 100k юзеров; блокировка мгновенно закрывает доступ (юзер получает 401/403); каждое действие в аудите.

### 4.3. Платежи и подписки (Billing)
- **Список подписок:** фильтр по статусу FSM (pending/active/grace/expired/cancelled/failed), плану, провайдеру, периоду; сортировка по дате/сумме.
- **Карточка подписки:** все поля (план, период, статус, суммы, даты started/expires/cancelled, auto-renew) + **audit-лог списаний** (`SubscriptionCharge`: успехи/отказы/возвраты, коды ошибок).
- **Действия:**
  - Отмена (support+) — `billing_service.cancel_subscription` (доступ до end-of-period).
  - Ручная выдача/продление (support+) — grant подписки без оплаты (гудвилл/триал/тех-компенсация), с указанием периода и причины.
  - **Возврат (superadmin)** — refund через провайдера, создаёт charge `refunded`.

**Acceptance:** отмена/выдача/возврат отражаются в статусе подписки и charge-логе; возврат идемпотентен; все операции с re-auth (возврат) и аудитом.

### 4.4. Статистика (Analytics)
- **Воронка:** signup → active(вошёл) → subscribed → churned за период.
- **Retention:** доля вернувшихся D1/D7/D30.
- **Drop-off по урокам:** где юзеры бросают (по `lesson_started` vs `lesson_completed` / `progress`).
- **Скилл:** распределение WPM/accuracy (гистограммы) по тирам.
- **Деньги:** динамика MRR, новые/отменённые подписки, decline rate (из charge-лога).

**Acceptance:** метрики считаются из `events`/`progress`/`attempts`/`subscriptions`, фильтр по периоду, тяжёлые запросы кэшируются.

## 5. Нефункциональные требования

- **Безопасность:** RBAC (3 роли); re-auth паролем на чувствительные операции; **аудит ВСЕХ действий админа** (кто/что/над кем/когда/IP-хеш); отдельный путь `/admin`; rate-limit на admin-эндпоинты. Имперсонация — короткий TTL, нельзя войти под другим админом.
- **152-ФЗ / приватность:** доступ к перс.данным юзеров логируется; IP хранится только хешем (`ip_hash`, уже так в `events`); имперсонация оставляет след.
- **Производительность:** список юзеров/подписок — пагинация + индексы; аналитика — кэш в Redis (TTL) для тяжёлых агрегатов.
- **i18n:** админка — **только ru** в v1 (внутренний инструмент).
- **Надёжность:** админка не должна ронять основной API (общий процесс, но изолированные эндпоинты/гейт).

---

## 6. Метрики успеха

- 0 прямых SQL-запросов в прод-БД руками для рутины (всё через админку).
- Время нахождения юзера по обращению < 30с.
- 100% возвратов/ручных операций с подписками — через админку (с аудитом), не через кабинет провайдера вслепую.
- Аналитик отвечает на «где отваливаются / какая конверсия» без обращения к разработчику.

## 7. Out of scope (v1)

- **Редактор контента уроков** — уроки сейчас статичные JSON в репо; онлайн-редактирование потребует переноса контента в БД (крупный отдельный эпик). Пока правки уроков — через git.
- **B2B / teams-панель** (организации, лицензии) — future.
- **2FA для админов** — future (на ревью Сергея-security; в v1 — re-auth паролем + аудит).
- **Экспорт отчётов (CSV/PDF)** — future.
- **Пуш-уведомления/тикет-система** внутри админки — интеграция позже.

## 8. Open questions

1. **Путь vs поддомен:** `/admin` (тот же домен) в v1 против отдельного `admin.typing-trainer.ru`. Рекомендация: `/admin` в v1, поддомен — позже (nginx allowlist по IP как усиление).
2. **2FA до prod?** Нужна ли обязательная 2FA для superadmin до выхода в прод — решение Сергея (security) при ревью.
3. **Кто первый superadmin:** бутстрап первой роли — через миграцию/CLI (см. TSD §2), не через саму админку.
4. **Гранулярность аналитик-доступа:** может ли analyst видеть точечные перс.данные (email конкретного юзера) или только агрегаты. Рекомендация: только агрегаты в v1.
